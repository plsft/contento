Deployment

Production deployment with Docker Compose, Cloudflare, and Nginx.

Docker Compose (Production)

The recommended production setup uses Docker Compose with three services: the Contento application, PostgreSQL 17, and Redis 7.

# docker-compose.yml (production)
version: "3.8"
services:
  app:
    build: .
    ports:
      - "5000:5000"
    environment:
      - DATABASE_CONNECTION_STRING=Host=postgres;Port=5432;Database=contento;Username=contento;Password=${DB_PASSWORD}
      - REDIS_CONNECTION_STRING=redis:6379
      - CONTENTO_ADMIN_PASSWORD=${ADMIN_PASSWORD}
      - GSC_CLIENT_ID=${GSC_CLIENT_ID}
      - GSC_CLIENT_SECRET=${GSC_CLIENT_SECRET}
      - ASPNETCORE_ENVIRONMENT=Production
      - ASPNETCORE_URLS=http://+:5000
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:5000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  postgres:
    image: postgres:17
    environment:
      POSTGRES_DB: contento
      POSTGRES_USER: contento
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U contento"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes:
      - redisdata:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

volumes:
  pgdata:
  redisdata:

Create a .env file alongside the compose file:

DB_PASSWORD=your-strong-database-password
ADMIN_PASSWORD=your-admin-password
GSC_CLIENT_ID=your-google-client-id
GSC_CLIENT_SECRET=your-google-client-secret

PostgreSQL 17 & Redis 7

If running outside of Docker Compose:

PostgreSQL 17

# Create the database
createdb -U postgres contento

# Verify version
psql -U postgres -c "SELECT version();"
# Should show PostgreSQL 17.x

Redis 7

# Verify Redis is running
redis-cli ping
# Should return PONG

# Verify version
redis-cli INFO server | grep redis_version
# Should show 7.x
Important: Redis is required for Contento pSEO (not optional). It powers the generation job queue, SSE progress streaming, and caching layer.

Cloudflare Configuration

Cloudflare is the recommended DNS and proxy provider for Contento pSEO.

1. Add your app domain

Add an A or CNAME record for your Contento application domain:

TypeNameContentProxy
Aappyour.server.ip.addressProxied (orange cloud)

2. Add pSEO subdomain support

For each project, add a CNAME record pointing to your app domain:

TypeNameContentProxy
CNAMEresourcesapp.contentocms.comProxied (orange cloud)

Alternatively, use a wildcard CNAME (*) if you plan to create many projects and want to avoid adding individual records.

3. SSL/TLS mode

Set SSL/TLS encryption mode to Full (strict) in the Cloudflare dashboard under SSL/TLS settings. This ensures end-to-end encryption between Cloudflare and your origin.

4. DNS proxy

Keep all records in Proxied mode (orange cloud icon) to benefit from Cloudflare's CDN caching, DDoS protection, and performance optimizations.

Nginx Reverse Proxy

If running Contento behind Nginx on a VPS or bare-metal server:

server {
    listen 443 ssl http2;
    server_name app.contentocms.com *.contentocms.com;

    ssl_certificate     /etc/ssl/certs/contentocms.pem;
    ssl_certificate_key /etc/ssl/private/contentocms.key;

    location / {
        proxy_pass http://127.0.0.1:5000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE support (for generation progress streaming)
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 3600s;
    }

    # Health check (no proxy overhead)
    location = /health {
        proxy_pass http://127.0.0.1:5000/health;
    }
}
SSE support: The proxy_buffering off directive is critical for the real-time generation progress SSE endpoint. Without it, Nginx buffers the event stream and clients see no updates until the connection closes.

Health Check

Contento exposes a health check endpoint at /health that returns:

# Healthy response (200 OK)
{
  "status": "healthy",
  "database": "connected",
  "redis": "connected",
  "uptime": "3d 14h 22m"
}

# Unhealthy response (503)
{
  "status": "unhealthy",
  "database": "disconnected",
  "redis": "connected"
}

Use this endpoint for load balancer health checks, Docker health checks, and uptime monitoring.

Database Migrations

Contento uses Bowtie for automatic schema management. On every startup, the application:

  1. Compares the current database schema against the expected schema derived from C# entity models.
  2. Generates and executes any necessary migration SQL (CREATE TABLE, ALTER TABLE, etc.).
  3. Logs all schema changes to the application log.

No manual migration commands are needed. Just deploy a new version and restart.

Seed Data

On first run (when the database is empty), Contento automatically seeds:

  • 150 niches across 8 categories with full context (audience, pain points, subtopics)
  • 10 content schemas (idea-list, comparison, how-to, best-of, beginner-guide, checklist, FAQ, alternatives, statistics, use-cases)
  • 1 admin user with the password from CONTENTO_ADMIN_PASSWORD

Seed data is idempotent — running the app again does not duplicate existing seed records.

Scaling Considerations

  • Horizontal scaling — Contento is stateless (all state lives in PostgreSQL and Redis). You can run multiple app instances behind a load balancer. Use sticky sessions for SSE connections.
  • Database — For 100k+ pages, consider PostgreSQL connection pooling (PgBouncer) and read replicas for analytics queries.
  • Redis — The default 256 MB memory limit handles up to ~50k queued jobs. Increase for very large generation batches.
  • AI rate limits — The concurrent workers setting in collections controls how many parallel AI requests are made. Tune this to stay within your AI provider's rate limits.
  • Disk — Generated HTML pages are stored in the database, not on disk. The main disk usage is PostgreSQL's data directory. Budget ~1 KB per page for storage estimates.
  • CDN caching — With Cloudflare proxied mode, generated pages are cached at the edge. Set appropriate Cache-Control headers to reduce origin load.
Tip: For a typical pSEO deployment with 10,000 pages, a single 2-CPU / 4 GB RAM instance with managed PostgreSQL and Redis is more than sufficient. Scale horizontally only when you exceed 100k pages or need higher generation throughput.
